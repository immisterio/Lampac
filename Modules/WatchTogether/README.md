# WatchTogether

**Совместный просмотр**: комнаты с синхронизацией состояния плеера (пауза, позиция, сезон/эпизод) между клиентами через **WebSocket (NWS)** и небольшой HTTP API для создания комнаты и JS-плагина.

## Назначение

- Пользователи с одинаковым `room_id` получают рассылку событий «кто что смотрит» и могут держать плеер в одном состоянии.
- Данные комнат хранятся в памяти (`RoomDb`); фоновая задача `GcTask` помогает убирать «зависшие» сущности.

## HTTP API

| Маршрут | Описание |
|---------|----------|
| `GET /watchtogether.js` | Плагин для Lampa с подстановкой `{localhost}` → базовый URL сервера. Кэш ~10 минут. |
| `GET /watchtogether/create` | Создание комнаты: query `title`, `tmdb_id`, `source`, `type`. Ответ: `{ "id": "<room_id>" }`. |
| `GET /watchtogether/info?id=` | Метаданные комнаты (название, tmdb, состояние воспроизведения, позиция, сезон/эпизод). |

## WebSocket: методы клиента

Клиент вызывает методы NWS (имена в нижнем регистре обрабатываются в `WsEvents`):

| Метод | Назначение |
|-------|------------|
| `watchtogether_join` | Вступить в комнату: аргументы `(roomId, uid)`. |
| `watchtogether_sync` | Синхронизация плеера: `(roomId, uid, state, position, season, episode)`. |
| `watchtogether_leave` | Покинуть комнату. |

Остальные методы с префиксом `watchtogether_` обрабатываются в том же обработчике (см. `WsEvents.cs`).

## Конфигурация

В стандартном `ModInit.cs` **нет** вызова `ModuleInvoke` — отдельной секции **`WatchTogether`** в `init.conf` по умолчанию нет; комнаты и логика полностью в коде модуля и NWS.

## Зависимости

- Требуется работающий **NWS** (общий WebSocket слой приложения).
- Плагин: `plugin.js` в каталоге модуля.

## Компоненты

| Файл | Роль |
|------|------|
| `WsEvents.cs` | Подписка на NWS, комнаты, рассылка синхронизации. |
| `WatchTogetherController.cs` | HTTP-маршруты и отдача JS. |
| `RoomDb.cs` | Хранение комнат и участников в памяти. |
| `GcTask.cs` | Фоновая очистка. |
